# Detection Rules

This document tracks detection rules for the local lab. Rules should remain
scoped to logs generated by this repository's local vulnerable app.

| Rule ID | Goal | Example signal | Severity | Status |
| --- | --- | --- | --- | --- |
| AUTH-BRUTE-FORCE-001 | Detect repeated failed login attempts against the local lab login flow | 5 or more `login_failure` events for the same `source_ip` and `username` within 5 minutes | Medium | Implemented |
| WEB-SQLI-PATTERN-001 | Detect SQL injection-like input patterns submitted to local lab routes | `suspicious_input` event with `signal` set to `sql_injection_like_pattern` | Medium | Implemented |
| WEB-XSS-PATTERN-001 | Detect XSS-like input patterns submitted to local lab routes | `suspicious_input` event with `signal` set to `xss_like_pattern` | Medium | Implemented |
| BAC-PRIV-ESC-001 | Detect admin-panel privilege escalation via broken access control | `admin_access` event with `signal` set to `broken_access_control_pattern` | High | Implemented |
| WEB-SSRF-INTERNAL-001 | Detect server-side fetches aimed at internal targets | `outbound_request` event with `signal` set to `ssrf_internal_target_pattern` | High | Implemented |
| WEB-BROKEN-ACCESS-001 | Detect possible broken access control behavior in local lab routes | `access_denied` or unexpected access events for fake restricted paths | High | Planned |

## AUTH-BRUTE-FORCE-001

Purpose:

- detect brute-force-like behavior in the local login lab

Input events:

- `event_type` equals `login_failure`
- `source_ip` identifies the local/private source address
- `username` identifies the fictional lab account being attempted
- `timestamp` is an ISO-8601 timestamp

The parser preserves the full raw event and ignores unknown additional fields
for this rule.

Threshold:

- 5 or more matching failures
- same `source_ip`
- same `username`
- within 5 minutes

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

Status:

- implemented in `detection-engine/detection_engine/rules.py`

## WEB-SQLI-PATTERN-001

Purpose:

- detect SQLi-like input signals submitted to the local search lab

Input events:

- `event_type` equals `suspicious_input`
- `signal` equals `sql_injection_like_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/search`
- `input_name` identifies the submitted field, currently `q`
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/search` accepts the SQLi-like input
- the app returns local demo results
- the app logs `reason` as `accepted_suspicious_input`

Secure-mode behavior:

- `/search` rejects the same SQLi-like input with HTTP `400`
- the app logs `reason` as `rejected_suspicious_input`

Matching logic:

- one matching `suspicious_input` event produces one finding

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally pattern-based and educational
- it should only be evaluated against this repository's local lab telemetry

Status:

- implemented in `detection-engine/detection_engine/rules.py`

## BAC-PRIV-ESC-001

Purpose:

- detect admin-panel privilege escalation through broken access control

Input events:

- `event_type` equals `admin_access`
- `signal` equals `broken_access_control_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/dashboard`
- `username` identifies the unauthorized viewer (often `guest`)
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/dashboard` trusts a client-supplied `role=admin` parameter and grants access
- the app logs `reason` as `broken_access_control_role_param` with the exploit signal

Secure-mode behavior:

- `/dashboard` ignores the `role` parameter and only honors a signed admin session
- the exploit attempt is denied with HTTP `403` and carries no exploit signal

Matching logic:

- one matching `admin_access` event with the exploit signal produces one finding
- legitimate admin sessions log `admin_access` without the signal and do not match

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally pattern-based and educational
- it should only be evaluated against this repository's local lab telemetry

Status:

- implemented in `detection-engine/detection_engine/rules.py`

## WEB-SSRF-INTERNAL-001

Purpose:

- detect server-side request forgery (SSRF) against the local fetch lab

Input events:

- `event_type` equals `outbound_request`
- `signal` equals `ssrf_internal_target_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/fetch`
- `target_url` is the user-supplied URL the server was asked to fetch
- `target_host` is the host parsed from `target_url`
- `timestamp` is an ISO-8601 timestamp

Internal-target signal:

The app flags a fetch as SSRF when the target URL either uses a non-http(s)
scheme (for example `file://`, `gopher://`, `dict://`) or resolves to a host in
a server-internal range:

- loopback (`127.0.0.0/8`, `::1`)
- private (`10/8`, `172.16/12`, `192.168/16`)
- link-local, including the cloud metadata endpoint `169.254.169.254`
- reserved or unspecified (`0.0.0.0`)
- internal hostnames such as `localhost`, `metadata`, `metadata.google.internal`

No real network request is ever made; the lab returns a simulated response so
the scenario stays deterministic and safe.

Vulnerable-mode behavior:

- `/fetch` retrieves any user-supplied URL, including internal targets
- the simulated metadata response returns fake credentials to make the lesson concrete
- the app logs `reason` as `fetched_internal_target` with the SSRF signal

Secure-mode behavior:

- `/fetch` enforces an http(s) allowlist and refuses internal targets with HTTP `400`
- the app logs `reason` as `blocked_internal_target`, still carrying the SSRF signal

Matching logic:

- one matching `outbound_request` event with the SSRF signal produces one finding
- the rule fires whether the attempt was served (insecure) or blocked (secure),
  because either is a detection-worthy SSRF attempt

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally pattern-based and educational
- it should only be evaluated against this repository's local lab telemetry

Status:

- implemented in `detection-engine/detection_engine/rules.py`
