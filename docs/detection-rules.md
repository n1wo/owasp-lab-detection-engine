## AUTH-BRUTE-FORCE-001

Purpose:

- detect brute-force-like behavior in the local login lab

Input events:

- `event_type` equals `login_failure`
- `source_ip` identifies the local/private source address
- `username` identifies the fictional lab account being attempted
- `timestamp` is an ISO-8601 timestamp

The parser preserves the full raw event and ignores unknown additional fields
for this rule.

Threshold:

- 5 or more matching failures
- same `source_ip`
- same `username`
- within 5 minutes

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

Status:

- implemented in `detection-engine/detection_engine/rules.py`

Purpose:

- detect SQLi-like input signals submitted to the local search lab

Input events:

- `event_type` equals `suspicious_input`
- `signal` equals `sql_injection_like_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/search`
- `input_name` identifies the submitted field, currently `q`
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/search` accepts the SQLi-like input
- the app returns local demo results
- the app logs `reason` as `accepted_suspicious_input`

Secure-mode behavior:

- `/search` rejects the same SQLi-like input with HTTP `400`
- the app logs `reason` as `rejected_suspicious_input`

Matching logic:

- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

- it should only be evaluated against this repository's local lab telemetry

- detect admin-panel privilege escalation through broken access control

Input events:

- `event_type` equals `admin_access`
- `signal` equals `broken_access_control_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/dashboard`
- `username` identifies the unauthorized viewer (often `guest`)
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/dashboard` trusts a client-supplied `role=admin` parameter and grants access

## WEB-SSRF-INTERNAL-001

Purpose:

- detect server-side request forgery (SSRF) against the local fetch lab

Input events:

- `event_type` equals `outbound_request`
- `signal` equals `ssrf_internal_target_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/fetch`
- `target_url` is the user-supplied URL the server was asked to fetch
- `target_host` is the host parsed from `target_url`
- `timestamp` is an ISO-8601 timestamp

Internal-target signal:

The app flags a fetch as SSRF when the target URL either uses a non-http(s)
scheme (for example `file://`, `gopher://`, `dict://`) or resolves to a host in
a server-internal range:

- loopback (`127.0.0.0/8`, `::1`)
- private (`10/8`, `172.16/12`, `192.168/16`)
- link-local, including the cloud metadata endpoint `169.254.169.254`
- reserved or unspecified (`0.0.0.0`)
- internal hostnames such as `localhost`, `metadata`, `metadata.google.internal`

No real network request is ever made; the lab returns a simulated response so
the scenario stays deterministic and safe.

Vulnerable-mode behavior:

- `/fetch` retrieves any user-supplied URL, including internal targets
- the simulated metadata response returns fake credentials to make the lesson concrete
- the app logs `reason` as `fetched_internal_target` with the SSRF signal

Secure-mode behavior:

- `/fetch` enforces an http(s) allowlist and refuses internal targets with HTTP `400`
- the app logs `reason` as `blocked_internal_target`, still carrying the SSRF signal

Matching logic:

- one matching `outbound_request` event with the SSRF signal produces one finding
- the rule fires whether the attempt was served (insecure) or blocked (secure),
  because either is a detection-worthy SSRF attempt

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally pattern-based and educational
- it should only be evaluated against this repository's local lab telemetry

Status:

- implemented in `detection-engine/detection_engine/rules.py`

## CONFIG-EXPOSURE-001

Purpose:

- detect a security misconfiguration where a debug/diagnostics endpoint is left
  enabled and discloses sensitive configuration

Input events:

- `event_type` equals `config_exposure`
- `signal` equals `config_exposure_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/debug`
- `exposed_keys` lists the configuration keys that were disclosed
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/debug` is enabled and dumps live configuration, including the signing
  `secret_key` and the lab credential set
- the app logs `reason` as `exposed_debug_config` with the exposure signal

Secure-mode behavior:

- `/debug` is disabled and returns HTTP `404` with no configuration
- the app logs `reason` as `debug_endpoint_disabled` and carries no signal, so
  the detection engine does not flag it

Matching logic:

  finding
- disabled-endpoint events log without the signal and do not match

Finding fields:

## CRYPTO-WEAK-001

Purpose:

- detect a cryptographic failure where a password is stored with a weak,
  unsalted hashing algorithm

Input events:

- `event_type` equals `credential_storage`
- `signal` equals `weak_password_hash_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/register`
- `algorithm` names the hashing algorithm used (e.g. `md5`)
- `salted` indicates whether a per-user salt was applied
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/register` stores the password as an unsalted MD5 digest
- the app logs `reason` as `weak_password_hash` with the exposure signal

Secure-mode behavior:

- `/register` derives a per-user salted PBKDF2-HMAC-SHA256 hash
- the app logs `reason` as `strong_password_hash` and carries no signal, so a
  properly hashed password is not flagged

Matching logic:

- one matching `credential_storage` event with the weak-hash signal produces one
  finding
- strong-hash events log without the signal and do not match

Finding fields:

## LOG-GAP-001

Purpose:

- detect a security logging and alerting failure where a privileged action is
  performed without an audit or alert record

Input events:

- `event_type` equals `sensitive_action`
- `signal` equals `logging_failure_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/admin/role`
- `action` names the sensitive operation (e.g. `role_change`)
- `target_user` identifies the account the action affected
- `audit_logged` / `alerted` indicate the (failed) audit posture
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/admin/role` performs the role change but its own auditing and alerting are
  off; no audit/alert trail is produced
- the app logs `reason` as `audit_logging_disabled` with the failure signal

Secure-mode behavior:

- `/admin/role` writes a full audit record and marks the action alerted
- the app logs `reason` as `audit_logged` and carries no signal, so an audited
  action is not flagged

Matching logic:

- one matching `sensitive_action` event with the failure signal produces one
  finding
- the external detection engine catches the gap the app's own monitoring missed
- audited actions log without the signal and do not match

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`

## INTEGRITY-DESERIALIZE-001

Purpose:

- detect a software/data-integrity failure where a serialized profile import
  trusts privileged fields controlled by the client

Input events:

- `event_type` equals `profile_import`
- `signal` equals `unsafe_deserialization_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/profile/import`
- `imported_keys` lists fields supplied in the serialized JSON profile
- `trusted_keys` lists fields accepted into the trusted profile object
- `privileged_keys` lists unexpected fields such as `role` or `feature_flags`
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/profile/import` accepts a JSON profile export and trusts every supplied key
- privileged client-controlled fields such as `role=admin` and
  `feature_flags=["admin_panel"]` are accepted into the trusted object
- the app logs `reason` as `trusted_serialized_privileged_fields` with the
  unsafe deserialization signal

Secure-mode behavior:

- `/profile/import` uses an allowlist for safe preference fields only
- privileged or unknown fields are rejected with HTTP `400`
- safe preference-only imports are accepted and log without the signal

Matching logic:

- one matching `profile_import` event with the unsafe deserialization signal
  produces one finding
- rejected secure-mode imports log without the signal and do not match

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally scoped to the local lab's profile import telemetry
- it should only be evaluated against this repository's local lab logs

Status:

- implemented in `detection-engine/detection_engine/rules.py`

## DESIGN-LOGIC-001

Purpose:

- detect an insecure-design flaw where checkout trusts a client-controlled
  final total instead of server-side business rules

Input events:

- `event_type` equals `business_action`
- `signal` equals `business_logic_abuse_pattern`
- `source_ip` identifies the local/private source address
- `request_path` identifies the local route, currently `/checkout`
- `action` identifies the workflow operation, currently `checkout`
- `expected_total` and `allowed_minimum` are server-calculated values
- `client_total` is the submitted final total
- `timestamp` is an ISO-8601 timestamp

Vulnerable-mode behavior:

- `/checkout` trusts the submitted final total
- a learner can submit `0.00` for an item with a server-side price
- the app logs `reason` as `trusted_client_controlled_total` with the business
  logic abuse signal

Secure-mode behavior:

- `/checkout` recalculates the minimum allowed total from server-side product
  data and discount rules
- impossible totals are rejected with HTTP `400`
- the rejected attempt still carries the signal so the SOC can alert on the
  attempted workflow abuse

Matching logic:

- one matching `business_action` event with the business logic abuse signal
  produces one finding
- valid checkout events log without the signal and do not match

Finding fields:

- `rule_id`
- `severity`
- `source_ip`
- `username`
- `event_count`
- `first_seen`
- `last_seen`
- `reason`

False-positive considerations inside the lab:

- this rule is intentionally scoped to the local lab checkout telemetry
- it should only be evaluated against this repository's local lab logs

Status:

- implemented in `detection-engine/detection_engine/rules.py`
